Critique Bloquant pour le trading live 3 points
1
Bot manager = simulateur random
api/bot_manager.py:144-168
La boucle _run() genere des trades aleatoires avec random.random(). Le Trader est instancie mais jamais utilise. En mode live, aucun vrai trade ne serait execute — le dashboard affiche des donnees fictives.
→ Brancher le Trader.execute_once() dans la boucle _run() avec un fetch de donnees reel via l'API Polymarket ou ccxt. Conserver le mode simulation comme fallback explicite.
2
Token ID en dur
api/bot_manager.py:135
token_id="placeholder" est code en dur. Aucun mecanisme pour selectionner un marche Polymarket reel. Meme si le Trader etait branche, il ne saurait pas sur quel marche operer.
→ Ajouter un selecteur de marche dans le dashboard et l'API (POST /api/bots/{key}/start avec un token_id en parametre). Proposer une liste des marches actifs via l'API Polymarket.
3
Pas de donnees live
bot/trader.py:106-122
run_loop() itere sur un DataFrame statique passe en parametre. Il n'y a aucun fetch de donnees temps reel — la boucle rejoue des donnees historiques ligne par ligne au lieu de consommer un flux live.
→ Implementer un fetch cyclique dans run_loop() : appeler downloader.fetch_ohlcv() ou polymarket_client.get_price() a chaque iteration avec un time.sleep(interval). Construire le DataFrame de facon incrementale.
Important Fiabilite compromise 4 points
4
Calcul PnL ambigu
bot/position_tracker.py:52-55
pnl = (exit - entry) / entry * size — size est en dollars, donc le resultat est un pourcentage multiplie par un montant. Mathematiquement correct, mais la semantique est confuse : le PnL est en dollars uniquement par coincidence.
→ Clarifier avec pnl = (exit - entry) * quantity ou quantity = size / entry est le nombre d'unites. Ajouter un commentaire expliquant l'unite du resultat.
5
init_db() appele a chaque trade
data/storage.py:66-67
Chaque appel a log_trade() execute init_db() qui cree une nouvelle connexion SQLite et relance CREATE TABLE IF NOT EXISTS. Pas de connection pooling — en haute frequence, risque de database is locked.
→ Initialiser la base une seule fois au demarrage. Utiliser un singleton de connexion avec threading.Lock ou passer a sqlite3.connect(check_same_thread=False) avec un pool.
6
Pas de validation PUT /api/settings
api/server.py:114-117
L'endpoint accepte n'importe quelle valeur : position_size: -999, stop_loss_pct: 500. Le modele Pydantic SettingsUpdate definit les types mais aucune borne. Un appel malveillant ou une erreur UI peut casser le risk manager.
→ Ajouter des validateurs Pydantic : Field(ge=0.01, le=1000) pour position_size, Field(ge=0.1, le=50) pour les pourcentages. Retourner 422 si hors bornes.
7
Mutation globale DRY_RUN
api/bot_manager.py:259
settings.DRY_RUN = self._settings["dry_run"] modifie une variable globale du module depuis un thread. Tous les threads de bots partagent cet etat sans synchronisation — race condition possible si un bot est en cours d'execution.
→ Passer dry_run en parametre d'instance au Trader et au PolymarketClient. Supprimer la mutation globale. Chaque bot lit sa propre config.
Moyen Design et robustesse 5 points
8
CORS grand ouvert
api/server.py:20-25
allow_origins=["*"] autorise n'importe quel site web a appeler l'API. En production, avec des cles privees Polygon en memoire, un site malveillant pourrait declencher des trades via le navigateur de l'utilisateur.
→ Restreindre a allow_origins=["http://localhost:1818"] ou rendre configurable via .env.
9
Scaler sans mecanisme de descente
incubation/scaler.py
L'echelle progressive monte de $1 a $100 quand les conditions sont remplies, mais ne redescend jamais. Si les performances se degradent apres un level-up, le bot continue a trader avec une taille de position elevee.
→ Ajouter un _evaluate_level_down() : si le win rate tombe sous 45% ou le profit factor sous 0.8, redescendre d'un niveau. Proteger le capital.
10
Stop-loss sur close uniquement
backtesting/engine.py:48-83
Le moteur de backtest verifie le stop-loss sur le close de la bougie, pas sur le low (pour un BUY) ou le high (pour un SELL). En realite, le stop serait touche intra-bougie — le backtest est donc trop optimiste.
→ Verifier le stop-loss sur curr["low"] pour les positions BUY et curr["high"] pour les SELL. Utiliser le prix du stop comme prix de sortie, pas le close.
11
CVD = approximation heuristique
strategies/cvd_strategy.py:24-28
Le volume delta est approxime par volume * (close - open) / (high - low). C'est une heuristique raisonnable en l'absence de donnees tick-level, mais la confidence du signal devrait refleter cette incertitude.
→ Appliquer un coefficient de reduction sur la confidence (ex: confidence *= 0.7) pour compenser l'approximation. Documenter la limitation dans la strategie.
12
Sharpe ratio mal annualise
api/bot_manager.py:215
arr.mean() / arr.std() * (252 ** 0.5) utilise le facteur 252 (jours de bourse) pour annualiser. Mais les trades arrivent toutes les 2-4 secondes en simulation — l'annualisation est incoherente et le Sharpe est surestime.
→ Calculer la frequence reelle des trades et utiliser sqrt(N) ou N est le nombre de periodes par an. Ou afficher le Sharpe non-annualise avec la mention "par trade".
Mineur Qualite de code 3 points
13
Pas de tests d'integration
tests/
Les tests unitaires couvrent les strategies, le backtest et le risk manager isolement. Mais il n'y a aucun test pour les routes FastAPI, le threading du BotManager, ni le flux complet signal → risk → order → position.
→ Ajouter des tests avec TestClient de FastAPI pour les endpoints. Tester le flux complet avec un PolymarketClient mocke en mode DRY_RUN.
14
Imports tardifs dans les fonctions
api/bot_manager.py:146, 213
import random et import numpy sont importes dans le corps de fonctions appelees frequemment. L'overhead est negligeable (Python cache les modules) mais c'est une mauvaise pratique qui nuit a la lisibilite.
→ Deplacer les imports en haut du fichier, avec les autres imports du module.
15
Pas de graceful shutdown
api/bot_manager.py
Quand le serveur est arrete (Ctrl+C), les threads daemon meurent sans cleanup. Les positions ouvertes ne sont pas sauvegardees, les ordres en cours ne sont pas annules sur Polymarket.
→ Intercepter SIGINT/SIGTERM pour appeler kill_all() + sauvegarder l'etat dans SQLite. Utiliser l'event shutdown de FastAPI.
